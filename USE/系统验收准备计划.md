## 📅 Day 5: 技术细节准备

### 🎯 目标: 准备技术问题回答

#### 1. 系统架构问题 (2小时)

**Q1: 为什么采用前后端分离架构?**
```
A: 
1. 职责分离: 前端专注UI交互,后端专注业务逻辑
2. 独立开发: 前后端可并行开发,提高效率
3. 技术选型灵活: 前端用Vue.js,后端用Spring Boot,各取所长
4. 易于维护: 模块化设计,便于后续扩展和维护
```

**Q2: 为什么算法服务要独立部署?**
```
A:
1. 语言隔离: 算法用Python实现,后端用Java,避免直接集成困难
2. 计算隔离: 算法计算密集,独立进程避免阻塞后端主线程
3. 可扩展性: 可独立扩展算法服务,支持负载均衡
4. 易于替换: 可方便地添加或替换算法,不影响后端
```

**Q3: 异步任务机制是如何实现的?**
```
A:
1. 前端提交任务后立即返回任务ID
2. 后端使用@Async注解将任务提交到线程池异步执行
3. 前端每2秒轮询任务状态
4. 任务状态持久化到数据库,服务重启不丢失
5. 避免HTTP超时问题 (大规模实例计算可能需要10分钟+)
```

**Q4: 系统的通信协议是什么?**
```
A:
1. 前端 ↔ 后端: RESTful API (HTTP/JSON)
2. 后端 ↔ 算法服务: RESTful API (HTTP/JSON)
3. 后端 ↔ 数据库: MyBatis ORM (JDBC)
4. 所有接口都是标准HTTP接口,易于测试和调试
```

#### 2. 算法实现问题 (2小时)

**Q5: GA+RL混合初始化是如何实现的?**
```
A:
1. 使用预训练的RouteFinder深度强化学习模型
2. 模型输入: MDVRP实例 (仓库、客户、车辆信息)
3. 模型输出: 高质量初始解
4. 混合策略: 20%种子解来自RL模型, 80%随机生成
5. 优势: 提高初始种群质量,加速收敛
```

**Q6: 跨仓库优化策略是什么?**
```
A:
1. 识别边界客户: 距离多个仓库都较近的客户
2. 重分配策略: 尝试将边界客户分配到不同仓库
3. 评估标准: 总成本是否降低
4. 实现方式: 在每次迭代中执行跨仓库优化
5. 效果: 改善多仓库协同效率
```

**Q7: 并行计算是如何实现的?**
```
A:
1. 使用Python multiprocessing模块
2. GA算法: 种群评估并行化 (每个个体独立评估)
3. ACO算法: 蚂蚁路径构建并行化 (每只蚂蚁独立构建)
4. 进程数: 根据CPU核心数自动调整 (默认6个进程)
5. 加速比: 在6核CPU上实现3-5倍加速
```

**Q8: 为什么PSO算法没有使用并行计算?**
```
A:
1. PSO算法的粒子更新需要全局最优信息
2. 粒子之间存在依赖关系,并行化收益不大
3. 当前实现采用单进程优化,计算效率已经很高
4. 未来可考虑采用异步PSO变体实现并行化
```

#### 3. 重规划功能问题 (2小时)

**Q9: 重规划算法的原理是什么?**
```
A:
1. 输入: 原始路径 + 阻塞路段列表
2. 识别受影响的路径: 包含阻塞路段的路径
3. 提取受影响客户: 这些路径上的所有客户
4. 设置临时仓库: 在受影响区域的中心位置
5. 重新规划: 使用GA算法为受影响客户重新规划路径
6. 输出: 新路径 + 成本对比
```

**Q10: 为什么要设置临时仓库?**
```
A:
1. 减少绕行距离: 避免从原仓库出发绕过阻塞路段
2. 降低成本: 临时仓库更接近受影响客户
3. 实际意义: 模拟应急调度中心或临时配送点
4. 位置选择: 受影响客户的几何中心
```

**Q11: 重规划的计算时间如何?**
```
A:
1. 通常只需要重新规划部分路径,计算量较小
2. 小规模重规划 (10-20个客户): 1-5秒
3. 中规模重规划 (50-100个客户): 10-30秒
4. 使用较小的种群和迭代次数,保证实时性
```

#### 4. 数据与实验问题 (1小时)

**Q12: 使用的测试数据集是什么?**
```
A:
1. Cordeau标准测试集 (P01-P23)
2. 国际公认的MDVRP基准测试集
3. 包含不同规模实例:
   - 小规模: P01-P07 (50-100客户)
   - 中规模: P08-P14 (100-200客户)
   - 大规模: P15-P23 (200-360客户)
4. 每个实例都有已知最优解(BKS)作为对比基准
```

**Q13: 实验结果如何评估?**
```
A:
1. 主要指标: Gap = (算法解 - BKS) / BKS × 100%
2. 次要指标: 计算时间、收敛速度
3. 每个实例运行3次取平均值
4. 对比基准: 已知最优解(BKS)
5. 统计分析: 平均Gap、最小Gap、最大Gap
```

**Q14: 为什么有些实例的Gap较大?**
```
A:
1. BKS是多年研究的最优结果,很难达到
2. 本研究重点是验证混合策略的有效性,而非追求最优解
3. 大规模实例 (P21-P23) 搜索空间巨大,Gap较大是正常的
4. 混合策略相对纯GA有改进,证明了方法的有效性
```
