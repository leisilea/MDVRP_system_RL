# P21 MDVRP求解总结

## 问题诊断

### 原始问题
用户报告："demand有问题，你是不是没有考虑车辆的车数问题"

### 根本原因
**误解了vehicle_capacity的含义**：
- ❌ 错误理解：`vehicle_capacity` = depot的总容量（所有车辆加起来）
- ✅ 正确理解：`vehicle_capacity` = 单辆车的容量

### 验证过程
通过测试RouteFinder在CVRP-100上的行为，发现：
```python
Actions shape: torch.Size([1, 109])  # 100个客户 + 9次返回depot
Number of depot visits (0s): 9       # 返回depot 9次 = 10条路线（10辆车）
```

**结论**：RouteFinder通过在actions序列中插入0（depot）来表示多条路线，每次返回depot就开始新的一条路线（新车辆）。

## 修复方案

### 代码修复
修改 `create_npz_for_depot()` 函数：

```python
# 修复前（错误）
capacity = depot['capacity'] / max_demand  # 把depot总容量当成单车容量

# 修复后（正确）
capacity_normalized = depot['capacity'] / max_demand  # 单车容量归一化
vehicle_capacity = np.array([[capacity_normalized]], dtype=np.float32)
```

### 验证结果
修复后，RouteFinder正确处理了多车辆：
- 理论最少需要：4辆车/depot（216需求 / 60容量 = 3.6 → 4辆）
- 实际使用：3-4辆车/depot（平均3.2辆）
- 总车辆数：28辆（9个depot）

## 问题分析

### 为什么Gap这么大？

1. **分割策略太简单**
   - K-Means和贪心都产生了完全均匀的分配（每depot 40客户）
   - BKS显示最优解是非常不均匀的（25-63客户）
   - P21数据结构对称（客户在同心圆上），导致两种方法结果相同

2. **失去全局优化**
   - 把MDVRP分解成9个独立的CVRP
   - 每个子问题的最优解加起来 ≠ 全局最优解
   - 客户-depot分配被预先固定，无法动态调整

3. **RouteFinder训练数据不匹配**
   - RouteFinder在CVRP-100上训练（100个客户，单depot）
   - P21是40个客户/depot，可能不在最优性能区间

## 改进方向

### 短期改进（容易实现）
1. ✅ **修复vehicle_capacity理解**（已完成）
2. 增加采样数量（从20增加到50或100）
3. 使用augmentation（dihedral8）
4. 调整分割策略（考虑容量约束）

### 中期改进（需要实验）
1. **迭代优化**：先分配 → 求解 → 根据结果重新分配 → 再求解
2. **容量感知分割**：分割时考虑车辆容量，避免过度不均衡
3. **边界客户重分配**：对靠近多个depot的客户尝试不同分配

### 长期改进（需要重新设计）
1. 使用专门的MDVRP算法（遗传算法、禁忌搜索、LKH3）
2. 训练专门的MDVRP深度学习模型
3. 混合方法：深度学习生成初始解 + 传统算法优化
